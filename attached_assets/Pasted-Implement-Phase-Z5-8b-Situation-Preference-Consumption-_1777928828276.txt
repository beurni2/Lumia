Known gap:
selectedSituations is currently not consumed by candidate generation, scenario selection, scoring, or ranking. This means the new required situations screen does not yet influence first-batch personalization.

Goal:
Wire selectedSituations into the existing generation/scoring path as an additive preference signal so first-session ideas better match the user's selected content lanes.

Hard rules:
- ZERO migrations.
- ZERO API breaking changes.
- ZERO validator loosening.
- ZERO broad architecture rewrite.
- Additive scoring/routing only.
- Preserve Hero Quality Ranker.
- Preserve Retention Novelty.
- Preserve willingness ranker.
- Preserve existing diversity/novelty/anti-copy gates.
- Do not hard-filter all non-selected situations; use soft weighting so quality does not collapse.
- Keep tests and TypeScript passing.

2. Add a selected-situation alignment score.
For each candidate, compute an additive score:
- strong match to selected situation: positive boost
- adjacent match: small positive boost
- unrelated: neutral
- clear mismatch only if many selected situations exist and candidate repeatedly falls outside them: small penalty

Suggested range:
- +4 strong match
- +2 adjacent match
- 0 neutral
- -1 or -2 repeated mismatch / low relevance

Keep this conservative enough that Hero Quality can still select a very strong candidate outside selected situations if needed.

3. Integrate with selection/ranking.
Add the situation alignment score into the existing selectionPenalty / IdeaScore / NoveltyContext path where taste calibration already influences ranking.
Do not create a separate engine.
Do not hard-filter unless candidate is unsafe or already rejected by existing gates.

4. First-session behavior.
For cold-start users with selectedSituations:
- selectedSituations should meaningfully influence the first batch.
- At least 1 of 3 ideas should strongly match a selected situation when candidate pool allows.
- Ideally 2 of 3 should match selected or adjacent situations.
- The batch should still preserve diversity and hero quality.

5. Returning-user behavior.
For returning users:
- selectedSituations should combine with ideator_signal and retention novelty.
- Actual behavior signals can override initial situation preferences over time.
- Do not trap the user forever in initial situations if exports/skips suggest taste shift.

6. Safety.
Private-data-adjacent situation lanes must remain safe:
- dating_texting must not require real private messages.
- creator_social analytics/comments should use fake/demo/cropped framing.
- work_school should not require confidential work/school data.
- money/bank adjacent examples should use fake/demo screens only if they arise from food_home/adulting style candidates.
