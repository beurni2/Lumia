Update Quick Tune calibration before closed beta.

Goal:
Improve onboarding personalization so Lumina captures actionable creator taste before first generation.

Decision:
Make the first 3 Quick Tune screens required:
1. Format / filming style
2. Tone / comedic voice
3. Situations / topic lanes

Make the 4th screen optional:
4. Opener style

Reason:
Format, tone, and situations are the strongest first-session calibration signals. Opener preference is useful but less essential and can also be learned from user behavior over time.

Hard rules:
- Prefer no migrations.
- Prefer existing calibration JSONB / existing save-load shape.
- No API breaking changes unless impossible.
- Preserve existing route response shape.
- Keep tests passing.
- Keep TypeScript clean.
- Preserve “Skip for now” only on optional opener screen.
- Required screens should not allow proceeding with zero selections unless there is an explicit “I’m not sure / surprise me” option.

Screen 1 — Required

Title:
What kind of video feels easiest to film?

Subtitle:
Pick up to 3 — this helps us give ideas you’d actually post.

Options:
1. Mini-stories
Subcopy:
“I said I’d be productive… then disappeared for 6 hours.”

2. Reaction moments
Subcopy:
“Opening something, seeing it, instantly regretting it.”

3. POV / relatable skits
Subcopy:
“POV: you’re acting busy but your brain left.”

4. A mix of everything
Subcopy:
“Surprise me and learn what I pick.”

Behavior:
- Required.
- Multi-select up to 3.
- If user selects “A mix of everything,” treat it as broad preference / balanced default.
- Do not allow Continue with zero selected.

Suggested mapping:
- Mini-stories → mini_story
- Reaction moments → reaction
- POV / relatable skits → pov_relatable_sketch / pov
- A mix of everything → balanced_mix

- Multi-select up to 3.

- User can continue with zero selected.

- Store skipped/empty opener preference explicitly if useful.

Suggested mapping:

- “This is where…” → curiosity_hook

- Calibration payload should include:
  selectedFormats
  selectedTones
  selectedSituations
  selectedOpeners
  openerSkipped boolean if useful
- Prefer mapping into existing taste calibration JSONB fields.
- If backend already has fields for formats/tones/hook styles, use them.
- If situations/topic lanes are new to the calibration payload, store safely in JSONB without migration if possible.
- Ensure backend generation can read topic/situation preferences if supported.
- If not currently consumed, store them now and report that backend consumption should be wired next.

The required screens should actually improve first generation quality. If selectedSituations are stored but not consumed by the engine, report that as a gap.
